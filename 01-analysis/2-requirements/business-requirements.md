# Бизнес-требования

> Этап 2. Почему это нужно бизнесу (цели, метрики, ограничения).

## Цели

| ID | Цель | Метрика успеха | Приоритет |
| :-- | :-- | :-- | :-- |
| BR-1 | Снизить нагрузку на владелицу мастерской за счёт автоматизации записи клиентов | Время на обработку одной записи снижается с ~15 мин (ручной диалог) до ~2 мин (самозапись) | Must |
| BR-2 | Исключить двойные брони и переполнение групп | 0 инцидентов двойной записи после запуска; овербукинг контролируется бэкендом | Must |
| BR-3 | Повысить удобство клиента и снизить барьер записи | Доля онлайн-записей среди всех записей ≥70% через месяц после запуска | Must |
| BR-4 | Дать клиентам прозрачность: видеть свободные слоты, программы, мастеров | Среднее число обращений в директ «а есть ли места» снижается на ≥80% | Should |

## Бизнес-ограничения

| ID | Ограничение | Влияние |
| :-- | :-- | :-- |
| BR-5 | Бюджет ограничен — начинаем с MVP | Скоуп сужен до клиентского приложения + API; оценки, лояльность, онлайн-оплата в Phase 2 |
| BR-6 | Срок — начало осеннего сезона (~2 месяца) | Жёсткая дата; функции разбиваются на MVP (must) и Phase 2 (should/could) |
| BR-7 | Оплата остаётся офлайн на старте | Приложение не интегрируется с платёжными системами в MVP |
| BR-8 | Существующая инфраструктура — black-box | Клиентское приложение полностью полагается на API бэкенда; не дублирует бизнес-логику |

## Риски

| ID | Риск | Митигация |
| :-- | :-- | :-- |
| BR-9 | Клиенты не перейдут на онлайн-запись, продолжат писать в директ | Доступность слотов 24/7, push-напоминания, удобный мобильный интерфейс |
| BR-10 | Поздние отмены оставят круги и глину простаивать | Политика 2 часов; статус late_cancel фиксируется для анализа админом |
