# Use Cases

> Этап 2. Сценарии использования системы (UC) — «кто и что делает».

## UC-1. Регистрация клиента

**Актор:** Клиент  
**Предусловие:** Клиент не авторизован  
**Триггер:** Клиент открывает приложение впервые или нажимает «Войти»

**Основной поток:**
1. Клиент вводит имя и номер телефона
2. Система отправляет SMS с OTP
3. Клиент вводит код из SMS
4. Система создаёт аккаунт и выдаёт JWT
5. Клиент попадает на экран списка слотов

**Альтернативы:**
- A1. Код не пришел — клиент запрашивает повторную отправку (лимит: 1 раз в 60 сек)
- A2. Код введён неверно 3 раза — система запрашивает новый код
- A3. Клиент уже зарегистрирован — система выполняет вход вместо регистрации

## UC-2. Просмотр и фильтрация слотов

**Актор:** Авторизованный клиент  
**Предусловие:** Клиент вошёл в приложение  
**Триггер:** Открытие экрана «Занятия»

**Основной поток:**
1. Система загружает слоты на ближайшие 7 дней
2. Клиент видит список: дата/время, программа, мастер, свободно/всего мест
3. Клиент применяет фильтры (дата, программа, мастер, только свободные)
4. Система обновляет список

**Исключения:**
- E1. Нет слотов в периоде — показывается empty state («Пока нет доступных занятий»)
- E2. Нет сети — показывается кэш с индикатором

## UC-3. Запись на занятие

**Актор:** Авторизованный клиент  
**Предусловие:** Клиент на карточке слота со свободными местами  
**Триггер:** Нажатие «Записаться»

**Основной поток:**
1. Клиент выбирает число мест (1–3)
2. Клиент выбирает для каждого места «свои инструменты» или «прокат»
3. Система валидирует доступность мест и прокатных наборов
4. Клиент подтверждает запись
5. Система создаёт бронь через API, возвращает `price_total`
6. Клиент видит экран подтверждения

**Исключения:**
- E1. Свободных мест или прокатных наборов недостаточно — показывается ошибка, кнопка заблокирована
- E2. Слот был отменён мастерской (410) — показывается сообщение с предложением выбрать другой слот

## UC-4. Отмена записи

**Актор:** Авторизованный клиент  
**Предусловие:** У клиента есть активная бронь  
**Триггер:** Нажатие «Отменить» в списке бронирований или деталях

**Основной поток:**
1. Клиент нажимает «Отменить»
2. Система запрашивает подтверждение
3. Клиент подтверждает
4. Система отправляет запрос на отмену
5. Бронь получает статус «Отменена» или «Поздняя отмена»

**Альтернативы:**
- A1. Ранняя отмена (≥2 ч) — места и прокат освобождаются
- A2. Поздняя отмена (<2 ч) — бронь фиксируется как `late_cancel`, места не освобождаются

## UC-5. Просмотр своих бронирований

**Актор:** Авторизованный клиент  
**Предусловие:** Клиент вошёл в приложение  
**Триггер:** Открытие экрана «Мои бронирования»

**Основной поток:**
1. Система загружает историю бронирований клиента
2. Клиент видит список: предстоящие сверху, прошедшие ниже
3. Каждая бронь: дата/время, программа, мастер, статус, число мест

## UC-6. Получение push-уведомления

**Актор:** Система (автоматически)  
**Предусловие:** Клиент дал разрешение на push  
**Триггер:** Наступило время напоминания или мастерская отменила занятие

**Основной поток:**
1. Система отправляет push на устройство клиента
2. Клиент видит уведомление:
   - Напоминание: «Завтра в 14:00 — Лепка для новичков, мастер Анна»
   - Отмена: «Занятие отменено: сломалась печь. Запись возвращена»
3. При тапе — переход в приложение на соответствующий экран
