# Примеры плохих промптов (антипаттерны)

> Чего **не стоит** делать при работе с документацией и кодом проекта.

---

## Расплывчатый запрос

```
Напиши документацию.
```
*Почему плохо: нет контекста о скоупе, аудитории, формате. Результат — много воды.*

---

## Запрос без указания формата

```
Сделай API.
```
*Почему плохо: непонятно, REST или GraphQL, какая версия, что входит.*

---

## Перекладывание решения на ИИ

```
Реши, что нам нужно в MVP.
```
*Почему плохо: ИИ не знает бизнес-контекста, бюджета, сроков. Решение принимает аналитик/заказчик.*

---

## Запрос на редактирование без контекста

```
Поправь файл.
```
*Почему плохо: непонятно, что именно и зачем.*

---

## Промпт без ограничений

```
Напиши всё, что можно, про систему.
```
*Почему плохо: результат будет неструктурированным, с перебором. Нужен скоуп.*

---

## Промпт без ссылки на контекст проекта

```
Напиши тест-кейсы.
```
*Почему плохо: ИИ не знает, какое приложение, какие границы скоупа, какие требования действуют. Получатся абстрактные шаблоны вместо проектных проверок.*

---

## Запрос на код без архитектурных ограничений

```
Напиши бэкенд для мастерской.
```
*Почему плохо: не указан стек (Go/Node/Python), нет данных о существующей инфраструктуре (black-box бэкенд), не определён контракт API. Результат — несовместим с проектом.*

---

## Перегруженный «всё-в-одном» запрос

```
Сделай анализ требований, спроектируй API, напиши код, напиши тесты и сделай UI.
```
*Почему плохо: каждая задача требует отдельного контекста и фокуса. Результат будет поверхностным по всем направлениям. Итеративный подход даёт качество.*

---

## Промпт без указания приоритетов

```
Добавь все функции из брифа.
```
*Почему плохо: в брифе есть Won't/Phase 2 (оценки мастеров, программа лояльности, онлайн-оплата). Без фильтра по MoSCoW — раздутый скоуп.*

---

## Промпт с противоречивыми указаниями

```
Сделай полное API, но не описывай поля — пусть разработчик сам додумает.
```
*Почему плохо: API-контракт — это источник истины для интеграции. Недоописанные поля приводят к багам на стыке клиент–бэкенд.*

---

**Правило:** Хороший промпт = контекст + задача + формат + ограничения + источники.
