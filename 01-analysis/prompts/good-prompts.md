## Создание тест-кейсов на основе аналитики

```
Ты находишься в контексте приложения pottery-workshop. Напиши тест-кейсы
на это приложение. Опирайся на аналитику: функциональные требования (FR),
use-cases (UC), нефункциональные требования (NFR) и API-контракты.

Структура:
- ID тест-кейса
- Название
- Предусловия
- Шаги
- Ожидаемый результат
- Трассировка к требованию

Покрой: авторизацию, слоты, бронирование, отмену, профиль, безопасность,
push-уведомления, доменную логику (unit), интеграцию с black-box бэкендом,
UI/UX, NFR, smoke-регресс.

В конце добавь матрицу трассировки: требование → список тест-кейсов.
```

## Промпт для debug: почему падает бронь

```
Клиент получает 409 при попытке записи на слот с free_seats=2, free_rental_kits=1.
Он запрашивает 2 места: одно с прокатом, одно со своими.
Почему это может падать? Проанализируй:
- формулу MaxSeats (FR-13)
- логику проверки free_rental_kits (FR-14)
- возможные состояния-гонки на стороне black-box бэкенда

Предложи сценарии воспроизведения.
```

---

**Правило:** Каждый хороший промпт содержит:
1. **Контекст** — где мы находимся, к какому проекту/файлу обращаемся
2. **Задачу** — что именно нужно сделать (конкретно)
3. **Формат** — в каком виде ожидается ответ (таблица, список, код)
4. **Ограничения** — скоуп, что исключить, приоритеты
5. **Источники** — на какие документы/файлы опираться
