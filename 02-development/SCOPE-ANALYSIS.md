## 2. Бизнес-ограничения (Business Constraints)

| ID | Ограничение | Влияние на скоуп | Хозяин решения |
|----|------------|------------------|----------------|
| **BR-5** | Бюджет ограничен — MVP | Сужение до must-have; Phase 2 отложена | Заказчик + продукт |
| **BR-6** | Срок ~2 месяца до сезона | Фокус на критичный путь; оценки и лояльность отброшены | Заказчик + продукт |
| **BR-7** | Оплата офлайн на старте | Нет интеграции платёжных систем | Бизнес + команда |
| **BR-8** | Существующая инфраструктура black-box | Go API не дублирует бизнес-логику | Архитектор |

### Следствия для архитектуры
- Go-сервис = **BFF (Backend for Frontend)**, не доменный сервис
- Не хранит primary данные слотов/мастеров — только кэш и mirror
- Не реализует бизнес-логику бронирования — проксирует на бэкенд
- Отвечает за: сессии, OTP, валидацию входа, агрегацию ответов

## 3. Доменные ограничения (Domain Constraints / Invariants)

### 3.1 Модель вместимости

| ID | Ограничение | Значение | Где проверяется |
|----|------------|----------|-----------------|
| **R-003** | capacity_cap по типу | Лепка ≤6, круг ≤10 | Бэкенд + БД CHECK + BFF валидация входа |
| **FR-13** | Макс. мест на бронь | `min(free_seats, capacity_cap, 3)` | **Бэкенд** (атомарно); BFF — pre-check |
| **FR-14** | Прокатный фонд | `rental_count <= free_rental_kits` | **Бэкенд** (атомарно); BFF — pre-check |
| **FR-15** | Запрет овербукинга | 0 двойных броней | **Бэкенд** (atomic) |
| **FR-16** | Отмена только целиком | Нет частичной отмены места/гостя | BFF (валидация запроса) + Бэкенд |

### 3.2 Модель времени и отмены

| ID | Ограничение | Значение | Где проверяется |
|----|------------|----------|-----------------|
| **FR-17** | Ранняя отмена ≥2 часов | Места освобождаются | **Бэкенд** (серверное время — источник истины) |
| **FR-18** | Поздняя отмена <2 часов | Статус `late_cancel`, места НЕ освобождаются | **Бэкенд** (решает по time) |
| **FR-35a** | История броней | Вся история, сортировка по `slot.start_at` | BFF (агрегация) |
| **NFR-17** | Время | UTC хранение, локальная TZ отображение | BFF (сериализация UTC); Клиент (отображение) |

### 3.3 Модель ценообразования

| ID | Ограничение | Формула | Где считается |
|----|------------|---------|---------------|
| **FR-45** | price_total от сервера | `price × seats_count + rental_price × rental_count` | **Бэкенд** — источник истины |
| — | Валюта | RUB | Конвенция |

### Ключевое архитектурное решение
> **R-004 (Граница интеграции):** Детали серверной реализации брони (транзакции, SLA, маппинг полей) — вне скоупа клиентского приложения. Существующий бэкенд выступает **black-box источником истины**; атомарность и гарантия «0 двойных броней» на его стороне.

## 4. Технические ограничения (NFR → Architectural Constraints)

### 4.1 Производительность

| ID | Требование | SLA | Кто отвечает |
|----|-----------|-----|-------------|
| **NFR-1** | Список слотов ≤500 мс (p95) | 100 RPS | Бэкенд + BFF (кэширование) |
| **NFR-2** | Создание брони ≤800 мс (p95) | — | **Бэкенд** (атомарность) |
| **NFR-3** | Cold start мобильного ≤3 сек | — | Мобильное приложение |

### 4.2 Надёжность

| ID | Требование | Как реализуется |
|----|-----------|---------------|
| **NFR-4** | Доступность ≥99.5% | Инфраструктура (не в скоупе API) |
| **NFR-5** | Offline: кэш слотов | Мобильное приложение (local DB / shared prefs) |

### 4.3 Безопасность

| ID | Требование | Кто реализует |
|----|-----------|---------------|
| **NFR-6** | TLS 1.2+, HTTPS only | Инфраструктура / Ingress |
| **NFR-7** | JWT: access 24ч, refresh 30дней | **BFF** (генерация/валидация) |
| **NFR-8** | Push-токены в Keystore/Keychain | Мобильное приложение |
| **NFR-9** | 152-ФЗ, не кэшировать чужие ПДн | **Бэкенд** + мобильное (запрет записи чужих контактов) |
| **NFR-10** | OTP: 6 цифр, TTL 5 мин, лимит 60 сек, 3 попытки | **BFF** (генерация, валидация, rate limit) |
| **NFR-11** | Блокировка после 5 неудач OTP в час | **BFF** (rate limiting в Redis) |

### 4.4 Масштабируемость

| ID | Требование | Кто отвечает |
|----|-----------|---------------|
| **NFR-12** | Горизонтальное масштабирование, stateless | Инфраструктура; BFF должен быть stateless (Redis для сессий) |

### 4.5 Совместимость

| ID | Требование | Кто отвечает |
|----|-----------|---------------|
| **NFR-13** | iOS 15+, Android 8.0+ | Мобильное приложение |
| **NFR-14** | 320 dp min width | Мобильное приложение |
| **NFR-15** | Тёмная тема | Мобильное приложение |

### 4.6 Локализация

| ID | Требование | Кто отвечает |
|----|-----------|---------------|
| **NFR-16** | Русский, ДД.ММ.ГГГГ, 24ч, ₽ | Мобильное приложение (отображение); BFF (формат API) |
| **NFR-17** | UTC хранение, локальная TZ | BFF (UTC); Мобильное (локальная TZ) |

### 4.7 Мониторинг

| ID | Требование | Кто отвечает |
|----|-----------|---------------|

## 5. Что НЕ входит в скоуп (вне границ)

### 5.1 Won't для MVP (отброшено)

| Функция | Почему вне скоупа | Когда |
|---------|------------------|-------|
| **Оценки мастеров** | Сужение MVP (BR-5, BR-6) | Phase 2 |
| **Онлайн-оплата** | BR-7: оплата наличными/переводом | Phase 2 |
| **Программа лояльности** | Не критично для сезона | Phase 2+ |
| **«Поделиться» занятием** | Согласовано с заказчицей (RR-D06) | Phase 2 |

### 5.2 Уже существует (не создаём)

| Функция | Существующая инфраструктура |
|---------|---------------------------|
| **CRUD расписания** | Интерфейс администратора |
| **CRUD мастеров** | Интерфейс администратора |
| **CRUD программ** | Интерфейс администратора |
| **Интерфейс мастера** | Существует отдельно |
| **Отмена слота мастерской** | Админка → push клиенту через BFF |

### 5.3 На стороне бэкенда (black-box)

| Функция | Почему там |
|---------|-----------|
| **Атомарное бронирование** | R-004: race conditions, транзакции |
| **Решение ранняя/поздняя отмена** | Нужно серверное время (источник истины) |
| **Подсчёт free_seats в реальном времени** | R-004: конкурентный доступ |
| **Обработка отмены мастерской** | Существующая админка |
| **Хранение primary данных** | Источник истины |

## 7. Требования к Go API (BFF) — сводный чеклист

### Must реализовать в BFF

| ID | Требование | Сложность |
|----|-----------|-----------|
| **FR-1/FR-2** | Регистрация/вход по телефону + SMS OTP | Средняя |
| **FR-43** | OTP: генерация, TTL 5 мин, 3 попытки, rate limit | Средняя |
| **NFR-10/NFR-11** | Rate limiting OTP (60 сек / номер, блокировка 30 мин) | Средняя |
| **FR-9** | Проксирование списка слотов + фильтры + пагинация | Низкая |
| **FR-38** | Валидация фильтров (date_from/to, program_type[], master_id[]) | Низкая |
| **FR-9a** | Проксирование карточки слота | Низкая |
| **FR-10/FR-11/FR-12** | Создание брони: валидация входа + проксирование на бэкенд | Средняя |
| **FR-16** | Запрет частичной отмены (валидация запроса) | Низкая |
| **FR-35a** | Проксирование истории броней + маппинг статусов | Низкая |
| **FR-47/FR-48/FR-49** | Профиль: GET/PATCH/DELETE + logout | Низкая |
| **FR-33** | Регистрация push-токенов, проксирование на push-сервис | Средняя |

### Must проксировать на бэкенд

| ID | Требование | Почему не в BFF |
|----|-----------|----------------|
| **FR-13** | `max_seats = min(free_seats, cap, 3)` | Динамика, race condition |
| **FR-14** | `rental_count <= free_rental_kits` | Cross-table atomic check |
| **FR-15** | 0 двойных броней | Атомарность транзакций |
| **FR-17/FR-18** | Решение ранняя/поздняя отмена | Нужно серверное время |
| **FR-45** | `price_total` | Источник истины |

### Must кэшировать в BFF

| Данные | TTL | Источник обновления |
|--------|-----|---------------------|
| Справочник программ | 1 час | Бэкенд (редко меняется) |
| Справочник мастеров | 1 час | Бэкенд (редко меняется) |
| Список слотов | 5 мин | Бэкенд (часто меняется) |
| Карточка слота | 1 мин | Бэкенд (free_seats меняется) |

## 8. Риски и компромиссы

| Риск | Причина | Митигация |
|------|---------|-----------|
| **BFF становится умным** | Дублирование бизнес-логики | Строгое следование R-004; вся логика в `domain/` packages без side-effects |
| **Несоответствие кэша** | free_seats устаревает | Короткий TTL (1–5 мин); stale data допустимо с empty state |
| **Rate limit обходится** | OTP flood | Redis + exponential backoff; блокировка по IP + phone |
| **JWT утечка** | Длинный refresh (30 дней) | Refresh rotation (опционально в Phase 2); secure storage на мобильном |
| **Push не доставлено** | Отказ от разрешений | Приложение работает без push; fallback на SMS (Phase 2) |
