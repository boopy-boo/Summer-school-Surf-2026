## 📋 РЕЗЮМЕ

Анализ выявил **значительное количество критических несоответствий** между требованиями, моделью данных, OpenAPI-спецификациями и мобильным ТЗ. Ключевые проблемы:
- **Противоречия в статусах брони** между FR, data-model и OpenAPI
- **Отсутствующие коды ошибок** в API, критичные для UX
- **Несоответствие модели цены** в OpenAPI vs FR-45
- **Пробелы в логике push-уведомлений** и offline-режима
- **Неявные зависимости** между удалением аккаунта и активными бронями
- **Размытые границы отмены** мастерской vs клиентской
- **Пропуски валидации** телефона и OTP на уровне API

## 1. 🚨 КРИТИЧЕСКИЕ ПРОТИВОРЕЧИЯ И НЕСООТВЕТСТВИЯ

### 1.1 Противоречие в статусах брони — DATA-MODEL vs OPENAPI vs ТЗ

| Источник | Статусы брони | Проблема |
|----------|---------------|----------|
| `data-model.md` (строка 63) | `active`, `cancelled`, `late_cancel`, `workshop_cancelled` | ✅ 4 статуса |
| `bookings/models.yaml` (строка 51) | `active`, `cancelled`, `late_cancel`, `workshop_cancelled` | ✅ 4 статуса |
| `SCR-005-my-bookings.md` (строка 60) | `active`, `cancelled`, `late_cancel`, `workshop_cancelled` | ✅ 4 статуса |
| `SCR-006-booking-details.md` (строка 36-39) | `cancelled`, `late_cancel` | ❌ Упоминает только 2 статуса отмены |
| `SCR-006-booking-details.md` | CTA «Отменить бронь» | ❌ Не описано поведение для `workshop_cancelled` |

**Импакт:** Экран деталей брони (SCR-006) не учитывает `workshop_cancelled`. Если клиент откроет детали брони, отменённой мастерской — будет показан CTA «Отменить бронь», который должен быть скрыт или disabled.

**Требуется:** Уточнить в SCR-006: для `workshop_cancelled` CTA «Отменить» скрыт, показывается бейдж «Отменено мастерской» с `cancellation_reason`.

---

### 1.2 Противоречие: «Состоялась» как статус — не существует в модели данных

- `SCR-005-my-bookings.md` (строка 50): бейдж «⚫ Состоялась» для прошедших занятий
- `data-model.md`: статусы брони — только 4 перечисленных выше
- Пройденное занятие со статусом `active` в data-model НЕ имеет статуса «Состоялась»

**Импакт:** Клиентская логика должна вычислять «Состоялась» по правилу: `status == active && slot.start_at < now()`. Это вычисляемое состояние, НЕ хранимый статус. В ТЗ это не задокументировано как вычисляемое поле.

**Требуется:** В SCR-005 и data-model явно зафиксировать, что «Состоялась» — клиентское вычисление, не статус API.

---

### 1.3 Противоречие модели цены: OpenAPI возвращает `price_total` на уровне слота, не брони

- `FR-45`: `price_total` — итоговая стоимость брони (read-only из API)
- `slots/models.yaml` (строка 104-109): `Slot.price_total` — итоговая стоимость брони, рассчитанная сервером... Формула: price × seats_count + rental_price × rental_count

**Проблема:** `price_total` в `Slot` schema описан как стоимость брони, но `Slot` — это не бронь. Бронь создаётся через `POST /bookings` и в `Booking` schema `price_total` тоже есть (строка 45-48).

**Вопрос:** Зачем `Slot` возвращает `price_total`?
- Если клиент выбирает 2 места + 1 прокат, сервер не знает выбор клиента на этапе `GET /slots/{id}`
- Формула в schema требует `seats_count` и `rental_count`, которых нет в запросе слота

**Импакт:** Неконсистентность API. Либо `price_total` в `Slot` лишний (должен быть только в `Booking`), либо schema некорректна.

**Требуется:** Удалить `price_total` из `Slot` schema или переименовать в `estimated_price` с дефолтом `seats_count=1, rental_count=0`.

---

### 1.4 Противоречие: `Slot.status = filled` — что это значит для бронирования?

- `slots/models.yaml` (строка 58-61): `filled` — мест нет
- `data-model.md` (строка 156): `filled` — все места заняты
- `SCR-003-slot-card.md` (строка 83): CTA disabled если `free_seats=0` ИЛИ `status=cancelled`

**Проблема:** Если `status=filled`, но `free_seats > 0` (возможно при отмене мастерской?), логика CTA сломается. Также `filled` не запрещает бронирование на уровне API — только по количеству мест.

**Требуется:** Зафиксировать инвариант: `status == filled` ⟹ `free_seats == 0`. Или убрать `filled` из API, полагаясь только на `free_seats`.

## 2. 🔴 ПРОБЕЛЫ В API-КОНТРАКТЕ (OpenAPI)

### 2.1 Отсутствует эндпоинт для push-токена

- `FR-33`: «Приложение регистрирует push-токен на устройстве»
- Ни в одном `api.yaml` нет эндпоинта для регистрации push-токена (POST /devices, POST /push/token и т.д.)

**Импакт:** Push-уведомления технически не реализуемы без контракта API.

**Требуется:** Добавить эндпоинт в API: `POST /push/register` с `{push_token, platform: ios|android}`.

---

### 2.2 Отсутствует `is_new` в `AuthResponse`

- `SCR-001-registration.md` (строка 101): «200, `is_new=false`» и «200, `is_new=true`»
- `auth/models.yaml` (строка 57-66): `AuthResponse` содержит `tokens` и `user`, **НЕТ поля `is_new`**

**Импакт:** ТЗ завязано на `is_new` для ветвления флоу (шаг 3 — имя для новых), а API его не возвращает.

**Требуется:** Добавить `is_new: boolean` в `AuthResponse`.

---

### 2.3 Отсутствует `first_booking` флаг в ответе создания брони

- `BS-002-booking-success.md` (строка 44): «При первой успешной записи (флаг `first_booking` в ответе или локально)»
- `bookings/models.yaml` (строка 23-62): `Booking` schema **НЕ содержит** `first_booking`

**Импакт:** Невозможно определить, показывать ли запрос push-разрешения на серверной стороне. Локальный флаг ненадёжен (другие устройства, переустановка).

**Требуется:** Добавить `first_booking: boolean` в `Booking` schema или использовать поле `created_at` профиля.

---

### 2.4 Отсутствует код 403 в OpenAPI для отмены брони

- `SCR-006-booking-details.md` (строка 38-39): «403: Уже отменена», «403: Занятие началось»
- `bookings/api.yaml` (строка 100-125): DELETE `/bookings/{id}` возвращает `200`, `400`, `401`, `404` — **НЕТ 403**

**Импакт:** API контракт не покрывает критические бизнес-ошибки отмены.

**Требуется:** Добавить 403 в спецификацию DELETE `/bookings/{id}` с описанием кейсов.

---

### 2.5 Отсутствует обработка 410 в GET `/slots/{id}`

- `FR-46`: повторная запись на отменённый слот запрещена → HTTP 410
- `slots/api.yaml` (строка 62-85): GET `/slots/{id}` возвращает 200, 401, 404 — **НЕТ 410**

**Импакт:** Если слот отменён мастерской, API returns 200 с `status=cancelled`, а клиент не понимает, что запись запрещена, пока не попробует `POST /bookings`.

**Требуется:** Уточнить: 410 возвращается только при `POST /bookings` или и при `GET /slots/{id}` тоже? Если нет — это допустимый флоу (смотреть можно, записаться нельзя), но должно быть явно описано.

---

### 2.6 `cancelled_at` в `Booking` schema — отсутствует в API

- `data-model.md` (строка 67): `Booking.cancelled_at: datetime?`
- `bookings/models.yaml` (строка 23-62): `Booking` schema **НЕ содержит** `cancelled_at`

**Импакт:** Клиент не может показать дату/время отмены в деталях брони.

**Требуется:** Добавить `cancelled_at` в `Booking` schema.

---

### 2.7 `AuthResponse` ссылается на `UserProfile`, но `UserProfile` не содержит `phone`

- `auth/models.yaml` (строка 65): `user: $ref: '../profile/models.yaml#/components/schemas/UserProfile'`
- `profile/models.yaml` (строка 17-25): `UserProfile` содержит `id`, `name`, `phone` — ✅ элемент есть

Проверка показывает `phone` присутствует. Но в `SCR-007-profile.md` написано: «телефон (только просмотр)» — в `UserProfileUpdate` (строка 27-33) нет `phone`. Это корректно: phone нельзя редактировать.

## 3. ⚠️ НЕОЧЕВИДНЫЕ ЗАВИСИМОСТИ И РИСКИ РЕГРЕССА

### 3.1 Удаление аккаунта (FR-48) — пробел в каскадной логике

- `FR-48`: «При удалении аккаунта: активные брони аннулируются и освобождают места»
- `profile/api.yaml`: DELETE `/profile` → 204

**Проблемы:**
1. Как клиент узнает, что его брони аннулированы? Push он уже не получит (токен удалён с устройства)
2. Что если удаление происходит < 2 часов до начала? Брони аннулируются — это ранняя или поздняя отмена?
3. Другие клиенты на этом слоте — получат ли они push об освобождении мест?

**Импакт:** Неопределённость в обработке активных броней при удалении аккаунта. Может привести к:
- Потере push-уведомлений об отмене
- Неконсистентному поведению при late_cancel границе
- Неясности в бизнес-логике: считается ли отмена по удалению аккаунта как late_cancel?

**Требуется:** Добавить в ТЗ и API:
- Поведение при удалении < 2 часов до занятия
- Push-уведомления другим клиентам об освобождении мест (если применимо)
- Локальное уведомление/алерт перед удалением: «У вас активные брони, они будут отменены»

---

### 3.2 Регистрация push-токена — курица и яйцо

- `FR-33`: Push-разрешение запрашивается «при первой успешной записи» (BS-002)
- `SCR-001-registration.md`: Токены сохраняются в Keychain, но push-токен не регистрируется при входе

**Проблема:** Пользователь может закрыть BS-002 или отказаться, а потом передумать. Как зарегистрировать push позже? Нет экрана настроек уведомлений.

**Импакт:** Потеря канала коммуникации. Если клиент отказался при первой записи — напоминания не придут никогда.

**Требуется:** Добавить в SCR-007 (Профиль) раздел «Уведомления» с возможностью включить/выключить push.

---

### 3.3 Offline-режим: кэширование и мутации

- `NFR-5`: «При недоступности сети приложение показывает кэшированные слоты»
- `foundations.md` (строка 145-147): «Мутации офлайн запрещены»

**Проблемы:**
1. Что происходит с кэшем броней при `DELETE /profile` (удаление аккаунта) offline? Клиент думает, что удалил аккаунт, но сервер ещё не получил запрос.
2. Как синхронизируется кэш при восстановлении сети? Нет описания механизма инвалидации.
3. Можно ли записаться на слот, который уже заполнился, если клиент смотрит кэш?

**Импакт:** Риск «фантомных броней» в offline. Клиент видит свободные места в кэше, но на сервере их уже нет.

**Требуется:** Добавить в ТЗ:
- Логику stale-индикатора (timestamp последнего обновления)
- Обязательный refresh перед мутациями (бронирование, отмена)
- Обработку 409 при бронировании из кэшированного состояния

---

### 3.4 Push-уведомление при отмене мастерской — target screen

- `FR-46`: push приходит, клиент тапает → «переход в приложение на соответствующий экран»
- `api-sequence.md` (строка 90): открывается SCR-006 (детали брони)

**Проблема:** Повторный тап на то же push или открытие приложения вручную — какой экран показывать? Нет deeplink-контракта.

**Импакт:** Невозможно реализовать навигацию по push без deeplink-схемы.

**Требуется:** Описать deeplink-формат: `glina://booking/{bookingId}` + экран назначения для каждого типа push.

---

### 3.5 Пагинация и разделение «Предстоящие / Прошедшие»

- `FR-35a`: «Разбиение на предстоящие и прошедшие выполняется на клиенте»
- `bookings/api.yaml` (строка 27): «Сортировка по `slot.start_at` убыванию»

**Проблема:** Если пагинация offset/limit и сортировка по `slot.start_at`, клиент может получить «дыру» в списке при одновременном создании новой брони:
1. Клиент загрузил брони offset=0, limit=20
2. Другая сессия (или тот же клиент) создала новую бронь
3. При scroll → offset=20 клиент пропустит брони, сдвинувшиеся из-за новой записи

**Импакт:** Пропажа броней в списке при инфинити-скролле.

**Требуется:** Заменить offset на cursor-based пагинацию (по `created_at` или `id`), или добавить pull-to-refresh с сбросом offset при каждом открытии.

## 4. 🔄 ПРОТИВОРЕЧИЯ МЕЖДУ ТЗ И ПЛАТФОРМЕННЫМИ ВОЗМОЖНОСТЯМЯМИ (iOS/Android)

### 4.1 Push-разрешение: iOS vs Android

- `BS-002-booking-success.md`: «При первой успешной записи — запрос push-разрешения»
- iOS: запрос push-разрешения ТРЕБУЕТ предварительной регистрации в APNs + entitlement. Нельзя запросить повторно после отказа (только через настройки системы).
- Android 13+: `POST_NOTIFICATIONS` runtime permission, можно запросить повторно.

**Проблема:** На iOS если клиент отказал — повторный запрос в BS-002 невозможен без переустановки приложения. ТЗ не учитывает платформенные ограничения.

**Требуется:** Добавить платформенную логику:
- iOS: показывать предварительный экран-просьбу (pre-permission) перед системным запросом
- Android: запрос runtime permission напрямую

---

### 4.2 Яндекс.Карты — SDK vs Static API

- `SCR-003-slot-card.md` (строка 77): «Карта — статичный превью Яндекс»
- `FR-44`: «Для отображения карты используется Яндекс.Карты»

**Проблемы:**
1. Яндекс Static Maps API требует API-ключ с квотами. Нет упоминания о ключе в NFR.
2. На iOS Apple Maps SDK предпочтительнее для performance и privacy. Требование «Яндекс.Карты» может быть избыточным.
3. Fallback: «текстовый адрес и координаты без интерактивной карты». Координаты пользователю зачем? Это технические данные.

**Требуется:** Уточнить в NFR:
- Какой именно API карт (Static API, JS SDK, нативный SDK)
- Политика ключа и квот
- На iOS допустимо ли использовать Apple Maps вместо Яндекс?

---

### 4.3 Keychain (iOS) vs Keystore (Android) — токены

- `NFR-8`: «Хранение push-токенов в защищённом хранилище устройства (Keychain / Keystore)»
- `SCR-001-registration.md` (строка 53): JWT токены в Keychain/Keystore

**Проблема:** Push-токен FCM/APNs — это не JWT. `NFR-8` говорит о push-токене, `SCR-001` — о JWT. Оба в защищённом хранилище?

**Уточнение:** На iOS Keychain — надёжно. На Android Keystore используется для ключей шифрования, не для строковых данных. JWT обычно хранят в EncryptedSharedPreferences.

**Требуется:** Конкретизировать механизм хранения для Android: `EncryptedSharedPreferences` с Keystore-backed master key.

---

### 4.4 SMS OTP на Android — авточтение и рекомендации Google

- `SCR-001-registration.md`: Поле кода — 6 ячеек, segmented
- Google рекомендации (2019+): использовать `SmsRetriever API` / `hint API` для авточтения SMS
- iOS 12+: `UITextField` с `.textContentType = .oneTimeCode` для автофилла из SMS

**Проблема:** ТЗ не упоминает автозаполнение кода из SMS. Это стандартное UX-ожидание на обеих платформах.

**Требуется:** Добавить AC: «Поле OTP поддерживает автофилл из SMS на iOS и Android».

## 6. 🎯 ПРОБЕЛЫ В ВАЛИДАЦИЯХ И ПОГРАНИЧНЫХ СЛУЧАЯХ

### 6.1 `max_seats` — где вычисляется?

- `FR-13`: `max_seats = min(free_seats, program.capacity_cap, 3)`
- `SCR-004-booking.md` (строка 73): «max = min(free_seats, 3)»

**Проблема:** В ТЗ экрана не учтён `program.capacity_cap`. Если `capacity_cap < free_seats < 3`, максимум должен быть `capacity_cap`, а не `free_seats`.

**Требуется:** Унифицировать формулу в SCR-004: `max_seats = min(free_seats, program.capacity_cap, 3)`.

---

### 6.2 Граница «ровно 2 часа» — UTC vs локальное время

- `FR-17`: «Граница ранней отмены — ровно за 2 часа до начала включительно»
- `FR-17` (примечание): «Время хранится и передаётся в UTC»
- Клиент видит локальное время, сервер принимает решение по UTC

**Проблема:** Пример: слот в 10:00 UTC+3 (07:00 UTC). Текущее время 08:00 UTC+3 (05:00 UTC). До начала 2 часа в локальном времени, но 2 часа в UTC. Граница «ровно 2 часа» неоднозначна.

**Требуется:** Уточнить: граница вычисляется сервером в UTC, клиент показывает приблизительное время и предупреждение за 2.5 часа.

---

### 6.3 Отмена в момент начала занятия

- `FR-16`: «Отмена возможна только до начала занятия»
- `FR-17`: «Граница ранней отмены — ровно за 2 часа»
- Что если `start_at == now()` (точно в момент начала)?

**Проблема:** Это «до начала» или «после»? Статус late_cancel или уже нельзя отменить?

**Требуется:** Зафиксировать: `start_at <= now()` → отмена невозможна (API returns 403).

---

### 6.4 `seats_count == 0` в `BookingCreateRequest`

- `BookingCreateRequest.seats` (строка 76): `minItems: 1, maxItems: 3`
- Массив `seats` определяет `seats_count` как `array.length`

**Вопрос:** Что если передать `seats: [{rental:false}, {rental:false}]` но `seats_count` на клиенте показывает 2, а сервер интерпретирует иначе?

**Требуется:** В API убрать избыточность: либо `seats_count` отдельно, либо только массив `seats`. Сейчас `seats_count` в `Booking` schema, но не в `BookingCreateRequest`.

---

### 6.5 Телефон в `verifyOtp`: уникальность и изменение

- `OtpVerifyRequest` (строка 33): требуется `name` при каждом verify
- Если клиент уже зарегистрирован и меняет имя — `name` передаётся заново? Перезаписывается?

**Проблема:** `verifyOtp` — идемпотентная операция? Если да, зачем `name` при каждом входе?

**Требуется:** Уточнить: `name` используется только при `is_new=true`. При `is_new=false` сервер игнорирует `name` или возвращает ошибку.

## 7. 🔗 НАВИГАЦИЯ И СОСТОЯНИЯ — ПРОБЕЛЫ

### 7.1 Отсутствует обработка 401 при истечении токена

- `NFR-7`: Access-token 24 часа, refresh — 30 дней
- Ни в одном экране ТЗ нет описания логики refresh:

**Проблема:** Если access-token истёк во время использования приложения:
- Какой экран показывать?
- Когда делать refresh — на каждом запросе или только при 401?
- Что если refresh тоже истёк?

**Требуется:** Добавить в foundations или отдельный документ логику:
- Silent refresh при 401 с `refresh_token`
- Если refresh failed → сброс сессии → SCR-001
- Экран-заглушка «Сессия истекла, войдите заново»

---

### 7.2 Глубина навигации и «Pop to Root»

- `navigation-map.md` (строка 80): BS-002 → «Pop to Root + Tab Switch» → SCR-005
- Какой именно root? SCR-002 или SCR-005?

**Проблема:** После записи переход на «Мои бронирования» сбрасывает стек навигации вкладки «Занятия». Если клиент потом нажмёт «Занятия» — он вернётся в корень, а не к слоту.

**Требуется:** Описать явно: удаление стека SCR-003→SCR-004 при переходе на SCR-005.

---

### 7.3 Нет экрана «Ошибка сети» как отдельного состояния

- `foundations.md`: Error state + кнопка «Обновить»
- Но нет ТЗ на модальный экран offline при попытке мутации

**Импакт:** Клиент нажимает «Подтвердить запись» offline — что показывать? Снек или bottom sheet?

**Требуется:** Добавить в SCR-004 обработку offline: снек + disabled CTA, или alert с предложением включить сеть.

## 8. ♻️ ПЕРЕИСПОЛЬЗУЕМЫЕ ЛОГИКИ — ДУБЛИРОВАНИЕ И ПРОБЕЛЫ

### 8.1 Логика «свободных мест» дублируется в 4 местах

- `FR-13`: `max_seats = min(free_seats, program.capacity_cap, 3)`
- `SCR-002-slot-list.md` (строка 134): CTA активен если `free_seats > 0`
- `SCR-003-slot-card.md` (строка 83): disabled если `free_seats=0`
- `SCR-004-booking.md` (строка 73): `max = min(free_seats, 3)`

**Проблема:** Если `program.capacity_cap` изменится на сервере, клиентская валидация в SCR-004 может быть несогласована.

**Требуется:** Вынести `max_seats` в API (в `Slot` schema) и использовать его единообразно.

---

### 8.2 Логика отображения статуса брони

- `SCR-005-my-bookings.md`: 4 статуса → 4 цвета
- `SCR-006-booking-details.md`: только активная/отменённая логика
- `data-model.md`: 4 статуса + переходы

**Проблема:** Цветовая кодировка не описана в foundations, а только в SCR-005. При добавлении нового статуса — нужно менять 2 экрана.

**Требуется:** Вынести цвета статусов в `foundations.md` как дизайн-токены.

---

### 8.3 Логика адреса мастерской

- `SCR-003`, `SCR-006`: адрес и карта
- `workshop_address` приходит в `Slot`

**Проблема:** Если разные слоты могут быть в разных мастерских (неясно из ТЗ), адрес в `Slot` корректен. Если мастерская одна — дублирование в каждом слоте.

**Требуется:** Уточнить: одна мастерская или несколько филиалов? Если одна — вынести адрес в отдельный API `/workshop`.
